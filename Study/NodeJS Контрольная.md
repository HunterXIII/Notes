# Ответы к тесту (по материалам лекций)

## 1. Что такое Node.js?

Node.js — это среда выполнения JavaScript-кода вне браузера, основанная на движке V8 от Google. Она позволяет использовать JavaScript не только для фронтенда, но и для создания серверных приложений.

## 2. Каковы основные архитектурные концепции Node.js?

- **Event-driven архитектура** — модель на событиях и коллбэках (асинхронное выполнение).
- **Неблокирующий I/O** — асинхронная обработка ввода-вывода (высокая производительность при множестве запросов).
- **Single-threaded модель** — однопоточное выполнение с циклом событий (event loop).
- **NPM** — менеджер пакетов и крупнейший репозиторий пакетов.

## 3. Какие основные команды используются при работе с NPM?

- `npm init` — создание `package.json`
- `npm install` — установка зависимостей
- `npm run` — запуск скриптов
- `npm publish` — публикация пакета

## 4. Какие системы модулей поддерживаются в Node.js?

- **CommonJS** — основной формат (`require` / `module.exports`).
- **ES Modules** — современный стандарт (`import` / `export`).

## 5. Как создать простой HTTP-сервер в Node.js (пример)?

```js
const http = require('http');
const server = http.createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'text/plain' });
  res.end('Hello World!');
});
server.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

## 6. Что такое Express.js?

Популярный, минималистичный и гибкий веб-фреймворк для JavaScript поверх Node.js. Даёт инструменты для веб-приложений и API: маршрутизация, обработка запросов, middleware. Устраняет ограничения «чистого» Node.js (ручные маршруты, нет middleware из коробки, повторяющийся код для заголовков и парсинга тел).

## 7. Что такое «промежуточное ПО» (middleware) в Express?

Функции, которые дополнительно обрабатывают запрос в конвейере (например, парсинг JSON, CORS, проверка JWT, логирование). Вызывают `next()`, чтобы передать управление дальше по стеку.

## 8. Как получить параметры маршрута (например, `/users/:userId`) в Express?

Через объект **`req.params`**: имена сегментов после `:` становятся ключами (например, `req.params.userId`).

## 9. Как получить параметры строки запроса (query string) в Express?

Через **`req.query`** — объект с параметрами после `?` в URL.

## 10. Что такое `express.Router()` и зачем он нужен?

Инструмент для создания **модульных, монтируемых** наборов маршрутов. Позволяет выносить логику в отдельные файлы, монтировать префикс через `app.use('/path', router)`, избегать дублирования префиксов и навешивать middleware только на нужный роутер (`router.use(...)`).

## 11. Что такое шаблонизаторы и как их использовать в Express?

Шаблонизаторы генерируют динамический HTML на сервере. В Express: **`app.set('view engine', ...)`**, **`app.set('views', ...)`**, ответ — **`res.render('шаблон', { данные })`**. Примеры из лекции: **EJS**, **Pug**, **Handlebars** (нужна установка пакета, для Handlebars — доп. настройка `app.engine`).

## 12. Что произойдет, если определить маршрут `/book/create` после маршрута `/book/:bookId`?

URL `/book/create` **совпадёт с** `/book/:bookId`, и сегмент `create` попадёт в **`bookId`**. Сработает **первый подходящий** маршрут — то есть обработчик **`/book/:bookId`**, а отдельный `/book/create` **не выполнится**. Чтобы обрабатывать `/book/create` отдельно, его нужно объявить **раньше**, чем `/book/:bookId`.

## 13. Что такое `req` и `res` в Express?

**`req`** — обогащённый объект **запроса**; **`res`** — обогащённый объект **ответа** (отправка данных, статусов, заголовков).

## 14. Как выглядит пример базового сервера на Express?

```js
const express = require('express');
const app = express();
app.get('/', (req, res) => {
  res.send('Hello World!');
});
app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

## 15. Как можно передать массив в строке запроса (query string) в Express?

Повторять имя параметра: **`?name=Tom&name=Bob&name=Sam`**.

## 16. Что такое Koa.js?

Веб-фреймворк нового поколения на Node.js от команды Express. Лаконичная архитектура, **`async/await`**, единый контекст **`ctx`**. Не «замена» Express, а параллельная ветка с другой философией.

## 17. Какие основные архитектурные принципы Koa?

- **Минимализм** — в ядре только основа, без встроенного JSON/static middleware.
- **Асинхронность на `async/await`**, без коллбэков в стиле Express.
- **Контекст `ctx`** — объединяет `req` и `res`.
- **Ориентация на веб** без лишнего.
- **Ошибки через исключения**, не через `next(err)`.
- Разработчик **сам подключает** нужные middleware.

## 18. Что такое объект контекста (`ctx`) в Koa?

Единый объект, объединяющий запрос и ответ Node.js: удобные поля для URL, метода, `query`, `params`, заголовков, тела ответа (`ctx.body`), статуса, редиректов и т.д.

## 19. Как обрабатываются ошибки в Koa?

Приложение — **EventEmitter**: **`app.on('error', handler)`** для логирования (не обязательно для ответа клиенту). Обычно в начале стека — **middleware с `try/catch`**: перехват, установка **`ctx.status`** и тела, затем **`app.emit('error', err, ctx)`**. Ошибки можно **бросать** из middleware.

## 20. Пример базового сервера в Koa

```js
const Koa = require('koa');
const app = new Koa();
app.use(async ctx => {
  ctx.body = 'Hello World!';
});
app.listen(3000, () => {
  console.log('Server running on port 3000');
});
```

## 21. Какую функцию выполняет `await next()` в Koa middleware?

Передаёт управление **следующему** middleware в цепочке; реализует **каскадную (луковичную)** модель: выполнение «вглубь» и возврат назад после завершения вложенных обработчиков.

## 22. Как отправить JSON-ответ в Koa?

Присвоить объект (или данные) в **`ctx.body`** — по аналогии с `res.json` в Express в таблице сравнения: **`ctx.body = { ... }`**. При необходимости задать **`ctx.status`**, тип через **`ctx.response.type`** / **`ctx.type`**.

## 23. В чем разница между `ctx.url` и `ctx.request.url` в Koa?

В лекции **`ctx.url`** указан как URL запроса; детали входящего запроса сосредоточены в **`ctx.request`** (в т.ч. **`ctx.request.url`**). На практике контекст даёт **сокращения** (`ctx.url`) для удобства; **`ctx.request`** — объект **запроса** Koa с полным доступом к свойствам запроса. Для ответа используется **`ctx.response`** / сокращения вроде **`ctx.body`**.

## 24. Как вручную выбросить ошибку с определенным статусом в Koa?

Использовать **`ctx.throw(404, 'Not Found')`** или **`throw`** ошибки с полем **`status`** (например, `error.status = 400`).

## 25. Для чего используется `router.all()` в koa-router?

Регистрирует обработчик, который срабатывает на **любой HTTP-метод** для заданного пути.

## 26. Что такое NestJS?

Прогрессивный фреймворк для эффективных и масштабируемых серверных приложений на Node.js с полноценной архитектурной основой и TypeScript по умолчанию.

## 27. Какие ограничения или недостатки у NestJS?

Высокая сложность для новичков (TypeScript, ООП, паттерны, DI/IoC); усложнённая отладка из-за слоёв; недостаточная документация по продвинутым темам; меньшая экосистема, чем у Express; повышенные требования к ресурсам; жёсткая структура даже для малых проектов; сложности с тестами (глубокие зависимости, микросервисы).

## 28. Для чего предназначен файл `main.ts` в проекте NestJS?

**Точка входа**: инициализация ядра Nest, создание экземпляра приложения, запуск прослушивания порта (по умолчанию 3000).

## 29. Какие HTTP-серверы поддерживает NestJS в качестве базовой платформы?

**Express** и **Fastify**.

## 30. Каково назначение декоратора `@Injectable()` в NestJS?

Помечает класс как провайдер, доступный **контейнеру внедрения зависимостей (DI)** для создания и внедрения в другие компоненты.

## 31. Для чего используется Guard (охранник) в NestJS?

Решает, **разрешить ли доступ** к маршруту (аутентификация, авторизация, роли). Основной метод — **`canActivate()`**; срабатывает **до** обработчика.

## 32. Каково типичное применение Interceptor в NestJS?

Логирование, **изменение структуры ответа**, кэширование, **измерение производительности** — логика **до и после** обработчика (`intercept()`).

## 33. Что такое DTO (Data Transfer Object) в NestJS?

Простые **классы**, описывающие структуру входящих данных; вместе с **ValidationPipe** и **class-validator** / **class-transformer** — автоматическая валидация и преобразование.

## 34. Как получить доступ к заголовкам HTTP-запроса в NestJS?

Через декоратор **`@Headers()`** (в лекции также упоминается доступ к запросу через **`@Request()`** и контекст выполнения в Guards).

## 35. В чем NestJS отличается от Express с точки зрения развертывания?

У Nest — структурированный **`ConfigService`** (`@nestjs/config`) для переменных окружения; из-за **TypeScript** нужен этап **сборки (транспиляции)**. У Express конфигурация часто **вручную** (например, **dotenv**), развёртывание **простого JavaScript** без обязательной компиляции.

## 36. Что такое Fastify?

Высокопроизводительный, лёгкий и расширяемый веб-фреймворк для Node.js с акцентом на скорость и низкие накладные расходы.

## 37. Каковы основные архитектурные особенности Fastify?

Высокая производительность; **архитектура плагинов** с **инкапсуляцией**; **JSON Schema** для валидации и сериализации; нативный **`async/await`**; логирование через **Pino**; удобный API и поддержка TypeScript; экосистема плагинов.

## 38. Для чего используются `reply.header()` и `reply.send()` в Fastify?

**`reply.header()`** — установка **HTTP-заголовков** ответа; **`reply.send()`** — отправка **тела ответа** (явное управление ответом при необходимости).

## 39. Ключевое преимущество JSON Schema в Fastify?

**Целостность данных** и **производительность**: схемы **компилируются** в эффективные функции валидации/сериализации.

## 40. Что такое система хуков (Hooks) в Fastify?

Механизм вставки логики на этапах жизненного цикла запроса (**`onRequest`**, **`preParsing`**, **`preValidation`**, **`preHandler`**, **`preSerialization`**, **`onSend`**, **`onResponse`** и т.д.) — глобально, в плагине или на маршруте.

## 41. Что такое инкапсуляция в архитектуре плагинов Fastify?

Каждый плагин по умолчанию в **изолированном контексте**: маршруты, хуки, декораторы не затрагивают «родительские» и «соседние» контексты; изменения наследуются **потомками** (вложенными плагинами/маршрутами).

## 42. Какая библиотека логирования используется в Fastify по умолчанию?

**Pino**.

## 43. Какой код статуса автоматически вернёт Fastify, если тело запроса не соответствует схеме валидации?

**400** (Bad Request) — в примере лекции явно указано при несоответствии **`request.body`** схеме.

## 44. Что такое `request.pathname` и чем он отличается от `request.url` в Fastify?

**`request.url`** — полный URL запроса (в примере с query: `/users/123?sort=name`); **`request.pathname`** — **путь без query** (например, `/users/123`).

## 45. Пример базового сервера на Fastify

```js
const fastify = require('fastify')({
  logger: true,
});
fastify.get('/', async (request, reply) => {
  return 'Hello World!';
});
const start = async () => {
  try {
    await fastify.listen({ port: 3000 });
  } catch (err) {
    fastify.log.error(err);
    process.exit(1);
  }
};
start();
```
