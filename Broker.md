# Архитектура и проектирование брокера сообщений  
  
## 1. Архитектурная схема системы  
![[Pasted image 20260315195337.png]]
Система реализована как **модульный монолит**. Это означает, что брокер представляет собой одно приложение (один процесс), но внутри него функциональность разделена на независимые логические модули.  
  
Такой подход выбран по нескольким причинам:  
  
- упрощает разработку и отладку  
- снижает сложность взаимодействия между компонентами  
- соответствует архитектуре большинства реальных брокеров сообщений  
- подходит для учебного проекта  
  
Каждый модуль отвечает за конкретную часть логики системы.  
  
Основные логические модули брокера:  
  
### Network Layer  
  
Отвечает за сетевое взаимодействие с клиентами.  
  
Основные задачи:  
  
- запуск TCP сервера  
- приём новых соединений  
- управление жизненным циклом соединения  
- чтение входящих данных от клиента  
- отправка сообщений клиентам  
  
Каждое подключение клиента обрабатывается в отдельной goroutine.  
  
---  
  
### Protocol Layer  
  
Отвечает за обработку протокола взаимодействия между клиентом и брокером.  
  
Задачи слоя:  
  
- декодирование входящих сообщений  
- проверка корректности запросов  
- преобразование сетевых сообщений во внутренние структуры брокера  
- маршрутизация команд в Broker Core  
  
Поддерживаемые команды:  
  
- `CONNECT`  
- `PUBLISH`  
- `SUBSCRIBE`  
- `DISCONNECT`  
  
---  
  
### Broker Core  
  
Центральный модуль системы.  
  
Отвечает за:  
  
- управление топиками  
- маршрутизацию сообщений  
- управление подписчиками  
- координацию работы остальных компонентов  
  
Broker Core хранит все топики и обеспечивает доступ к ним.  
  
---  
  
### Topic Manager  
  
Управляет топиками и их состоянием.  
  
Функции:  
  
- создание новых топиков  
- получение существующих топиков  
- хранение списка подписчиков  
- передача сообщений в очередь топика  
  
Топики хранятся в памяти брокера.  
  
---  
  
### Queue Manager  
  
Отвечает за очереди сообщений.  
  
Каждый топик имеет собственную очередь сообщений.  
  
Основные задачи:  
  
- добавление сообщений в очередь  
- извлечение сообщений из очереди  
- поддержание порядка доставки сообщений  
  
Тип очереди по умолчанию:  
  
- FIFO (First In First Out)  
  
---  
  
### Delivery Engine  
  
Отвечает за доставку сообщений подписчикам.  
  
Алгоритм работы:  
  
1. сообщение добавляется в очередь топика  
2. delivery engine получает сообщение  
3. сообщение отправляется всем подписчикам топика  
4. ожидается подтверждение доставки (если используется)  
  
Гарантия доставки:  
  
- **at-least-once**  
  
Это означает, что сообщение может быть доставлено более одного раза, но не будет потеряно.  
  
---
### Storage Layer  
  
Отвечает за персистентность сообщений.  
  
Основные задачи:  
  
- запись сообщений на диск  
- восстановление состояния после перезапуска брокера  
  
Используется **append-only лог**.  
  
Сообщения записываются последовательно в файл: `data/messages.log`
  
При запуске брокера лог читается и используется для восстановления состояния.  
  
---  
  
### Monitoring Layer  
  
Обеспечивает сбор и экспорт метрик.  
  
Метрики доступны через HTTP endpoint: `/metrics`
  
Метрики предназначены для интеграции с системой мониторинга Prometheus.  
Примеры метрик:  
  
- количество опубликованных сообщений  
- количество доставленных сообщений  
- количество активных подписчиков  
- количество топиков  
- размер очередей  
  
---  
  
## 2. Описание структуры данных  
  
Брокер оперирует несколькими основными сущностями:  
  
- Message  
- Topic  
- Queue  
- Subscriber  
- Broker  
  
---  
  
### Message  
  
Сообщение — основная единица передачи данных.  
  
Структура сообщения:  
  
```go  
type Message struct {  
    ID string  
    Topic string  
    Payload []byte  
    Timestamp int64  
    TTL int64  
}
```
### Описание полей  
  
| Поле      | Описание                           |
| --------- | ---------------------------------- |
| ID        | уникальный идентификатор сообщения |
| Topic     | название топика                    |
| Payload   | полезная нагрузка                  |
| Timestamp | время публикации                   |
| TTL       | время жизни сообщения              |
  
Поле TTL используется для удаления устаревших сообщений.  
  
---  
  
### Topic  
  
Топик — логическая категория сообщений.  
  
Каждый топик содержит очередь сообщений и список подписчиков.  
  
```go  
type Topic struct {  
    Name string  
    Queue *Queue  
    Subscribers map[string]*Subscriber  
}
```
Описание:

|Поле|Описание|
|---|---|
|Name|имя топика|
|Queue|очередь сообщений|
|Subscribers|список подписчиков|
Основные операции:
- добавление сообщения (push)
- извлечение сообщения (pop)
- просмотр первого сообщения (peek)

Сообщения обрабатываются в порядке FIFO.

---
### Subscriber

Subscriber представляет клиента брокера, который получает сообщения.
```go
type Subscriber struct {  
    ID         string  
    Connection net.Conn  
    Topics     map[string]bool  
}
```

Описание:

|Поле|Описание|
|---|---|
|ID|идентификатор клиента|
|Connection|TCP соединение|
|Topics|список подписок|

---
### Broker

Центральная структура системы.

```go
type Broker struct {  
    Topics map[string]*Topic  
}
```
Broker хранит все топики и управляет их жизненным циклом.

---
### Формат хранения сообщений

Сообщения сохраняются в файл журнала.

Формат записи:

```
timestamp|message_id|topic|payload
```

Пример:

```
1710412000|msg1|orders|new order  
1710412010|msg2|orders|cancel order
```
Использование append-only лога позволяет:
- быстро записывать сообщения
- упростить восстановление состояния
- минимизировать накладные расходы хранения

## 3. Технический стек  
  
### Язык программирования  
  
**Go**  
  
Причины выбора:  
  
- встроенная поддержка конкурентности (goroutines, channels)  
- высокая производительность  
- простота разработки сетевых сервисов  
- богатая стандартная библиотека  
- широкое использование в инфраструктурных системах  
  
---  
  
### Протокол взаимодействия  
  
**MQTT**  
  
Причины выбора:  
  
- нативная поддержка pub/sub модели  
- простая и лёгкая спецификация  
- небольшие сетевые пакеты  
- подходит для систем обмена сообщениями  
  
MQTT широко используется в системах обмена сообщениями и IoT.  
  
---  
  
### Персистентность  
  
**Файловая система**  
  
Используется append-only лог сообщений.  
  
Причины выбора:  
  
- высокая скорость записи  
- простая реализация  
- отсутствие необходимости использовать внешнюю базу данных  
- подходит для задач брокера сообщений  
  
---  
  
### Мониторинг  
  
**Prometheus**  
  
Метрики экспортируются через HTTP endpoint: `/metrics`

---  

### Контейнеризация  
  
**Docker**  
  
Планируется использование:  
  
- Dockerfile для сборки приложения  
- docker-compose для запуска системы  
  
---  
  
## 4. План реализации  
### Этап 1. Базовый сервер  
  
- запуск TCP сервера  
- обработка подключений клиентов  
- чтение данных из соединения  
- отправка данных клиенту  
  
---  
  
### Этап 2. Реализация протокола  
  
Обработка команд:  
  
- CONNECT  
- PUBLISH  
- SUBSCRIBE  
- DISCONNECT  
  
---  
  
### Этап 3. Broker Core  
  
- структура Broker  
- хранение топиков  
- создание и получение топиков  
  
---  
  
### Этап 4. Реализация топиков и очередей  
  
- Topic Manager  
- Queue Manager  
- публикация сообщений  
- добавление сообщений в очередь  
- обработка очередей  
  
---  
  
### Этап 5. Delivery Engine  
  
- отправка сообщений подписчикам  
- обработка отключений клиентов  
- управление подписками  
  
---  
  
### Этап 6. Персистентность  
  
- запись сообщений в log файл  
- восстановление состояния при запуске брокера  
  
---  
  
### Этап 7. SDK для Go  
  
Основные функции:  
  
- подключение к брокеру  
- публикация сообщений  
- подписка на топики  
  
---  
  
### Этап 8. Метрики  
  
- endpoint `/metrics`  
- экспорт метрик для Prometheus  
  
---  
  
### Этап 9. Web UI  
  
- отображение топиков  
- количество сообщений  
- активные подписчики  
- базовые метрики брокера  
  
---
